1. Why does multithreading not provide a significant speed advantage here?
Multithreading does not provide a large speed improvement because CPython uses the Global Interpreter Lock (GIL), which allows only one thread to 
execute Python bytecode at a time.
This task is mainly CPU-bound. Even though multiple threads are created, they cannot run Python code truly in parallel on multiple cores. The small
speed difference observed is likely due to measurement noise and the fact that NumPy internally uses optimized native code.

2. What exactly does the GIL do?
The Global Interpreter Lock is a mutex inside CPython that ensures only one thread executes Python bytecode at any given moment.
Its main purposes are:
.Protect Python’s memory management
.Prevent race conditions inside the interpreter
.Simplify CPython’s implementation
However, the downside is that it prevents true parallel execution of CPU-bound Python threads.

3. Are there programming languages or Python implementations without GIL?
Yes.
Python implementations without GIL include:
.Jython
.IronPython
Other programming languages such as C++, Java, and Go also do not have a GIL and allow true multithreaded parallelism.

4. How can I still achieve parallelism for CPU-intensive tasks?
To achieve real parallelism in Python for CPU-bound workloads, you can:
.Use multiprocessing (separate processes bypass the GIL)
.Use libraries that release the GIL internally (e.g., NumPy, some C extensions)
.Write performance-critical parts in C/Cython
.Use parallel computing frameworks (e.g., joblib, Dask)
The most common and straightforward method in standard Python is multiprocessing, since each process has its own interpreter and its own GIL.
